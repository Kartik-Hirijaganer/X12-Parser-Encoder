Problem

A non-technical colleague in DC home healthcare needs to perform bulk Medicaid eligibility checks (270/271 transactions) against Gainwell Technologies' DC MMIS system. Currently this is manual and error-prone. There is no lightweight, open-source, Python-native tool for constructing and parsing X12 270/271 eligibility transactions that complies with the DC Medicaid Companion Guide (v1.4, January 2026, 005010X279A1).

Solution: Two Deliverables

📦
x12-edi-tools PyPI Library

A pip-installable Python library for parsing, encoding, and validating X12 transactions. Generic X12-first, with the first production-ready profile pack for 270/271 005010X279A1 (DC Medicaid / Gainwell).

pip install x12-edi-tools
🌐
Eligibility Workbench Web App

A stateless web application (FastAPI + React) wrapping the library with a drag-and-drop UI for non-technical users doing bulk eligibility workflows. No database, no session storage.

docker compose up

Assumptions

  1. Python 3.11+ (Pydantic v2 compatibility, modern type hints)
  2. Only 270/271 transaction types in v1.0 (extensible for 837/835 later)
  3. DC Medicaid-specific: ISA08/GS03 = DCMEDICAID, no dependent loops (all members are primary subscribers)
  4. Stateless web app — no database, no user accounts, no sessions, no file retention
  5. Near-zero deployment cost (Vercel/Render for demos; Cloud Run for production PHI)
  6. Companion guide at metadata/full_text.txt is dev reference only — NEVER published
  7. MIT license, open source from day one
  8. PHI Safety: No PHI in logs, no PHI in test fixtures, no server-side file retention. All test fixtures must be synthetic.
  9. UI import is strictly template-based (canonical columns), not heuristic guessing
  10. Library is generic X12-first; DC Medicaid rules live in a pluggable payer profile pack

System Architecture

The system is a monorepo with two primary deliverables: a reusable PyPI library (packages/x12-edi-tools) and a deployable web application (apps/) that consumes the library.

Figure 1 — System Architecture: Monorepo with reusable library + deployable web application

Parser Pipeline Architecture

The parser operates as a four-stage pipeline transforming raw X12 strings into fully-typed Pydantic model trees.

Figure 2 — Four-Stage Parser Pipeline: Raw X12 → ISA → Tokens → Segments → Nested Model Tree

Encoder Architecture

The encoder reverses the parser pipeline, transforming typed Pydantic models back into valid X12 ANSI strings with auto-computed envelope counts.

segment_encoder.py

Calls segment.to_elements(), joins with element separator, trims trailing empties, appends segment terminator.

isa_encoder.py

Special handling for ISA fixed-width: pads each field to exact width, outputs exactly 106 characters.

x12_encoder.py

Walks Interchange tree depth-first. Auto-computes SE01 (segment count), GE01 (transaction count), IEA01 (group count).

💡
Auto-compute counts: If a user builds an Interchange programmatically, they might get SE01/GE01/IEA01 wrong. Computing them during encoding prevents SNIP Level 3 errors. Original parsed values remain on the model for inspection.

Validator Architecture

Five generic SNIP validation levels plus pluggable payer-specific profile packs. All results normalized into a common ValidationError structure with severity, plain-English messages, and actionable fix suggestions.

Figure 3 — Validator Architecture: 5 SNIP Levels + Pluggable Payer Profile Packs

Transaction Models

Transaction270

Eligibility inquiry. Contains ST, BHT, Loop2000A (with nested 2000B, 2000C), SE. Each Loop2000C has EQ segments for service type inquiries.

Transaction271

Eligibility response. Same structure but Loop2100C may include N3/N4/AAA, and Loop2110C contains EB segments with benefit info + optional Loop2120C entities.

Library: Convenience API (Agent-Friendly)

High-level functions for zero-config usage. Designed for LLM agents and non-expert callers.

import x12_edi_tools

# Import: auto-detect format, auto-correct dates/names/IDs
patients = x12_edi_tools.from_csv("patients.csv")
patients = x12_edi_tools.from_excel("patients.xlsx")

# Generate: one call, all defaults filled
interchange = x12_edi_tools.build_270(patients, profile="dc_medicaid")

# Validate: plain-English results
result = x12_edi_tools.validate(interchange, profile="dc_medicaid")
print(result.human_readable_summary())

# Encode + write
x12_content = x12_edi_tools.encode(interchange)
Path("output.x12").write_text(x12_content)

# Parse 271 responses: structured eligibility, not raw X12
eligibility = x12_edi_tools.read_271("response.x12")
df = eligibility.to_dataframe()  # optional pandas integration

SubmitterConfig

Provider/submitter identity and envelope settings. Passed from the frontend via API request body — the backend never reads from server-side config files for submitter identity.

class SubmitterConfig(BaseModel):
    # Group 1: Submitter Identity
    organization_name: str                  # NM103 in 2100B
    provider_npi: str                       # NM109 in 2100B
    provider_entity_type: str = "2"         # NM102 in 2100B
    trading_partner_id: str                 # ISA06, GS02
    provider_taxonomy_code: str | None = None  # PRV03

    # Group 2: Payer/Receiver (from profile, overridable)
    payer_name: str                          # NM103 in 2100A
    payer_id: str                            # NM109 in 2100A
    interchange_receiver_id: str            # ISA08
    receiver_id_qualifier: str = "ZZ"       # ISA07

    # Group 3: Envelope
    sender_id_qualifier: str = "ZZ"         # ISA05
    usage_indicator: str = "T"             # ISA15 (T=Test, P=Production)
    acknowledgment_requested: str = "0"    # ISA14

    # Group 4: Transaction Defaults
    default_service_type_code: str = "30"   # Health Benefit Plan Coverage
    default_service_date: str | None = None  # None = today
    max_batch_size: int = 5000              # DC Medicaid hard limit

Configuration System

The web UI includes a Settings panel where the user configures their provider identity, target payer, and envelope defaults before uploading files. Settings persist in browser localStorage (no server-side storage).

Configuration Groups

Group 1: Submitter / Provider Identity

Who you are — set once, used for every 270 file generated.

  • Organization Name (required)
  • Provider NPI (required, Luhn-validated)
  • Provider Entity Type (default: 2)
  • Trading Partner ID (required)
  • Taxonomy Code, Contact Info (optional)
Group 2: Payer / Receiver Identity

Who you're sending to — auto-populated from payer profile.

  • Payer Profile selection (drives defaults)
  • Payer Name: "DC MEDICAID"
  • Payer ID: "DCMEDICAID"
  • Interchange Receiver ID: "DCMEDICAID"
  • Receiver ID Qualifier: "ZZ"
Group 3: Envelope Defaults

X12 envelope settings — rarely change but must be configurable.

  • Sender ID Qualifier (default: ZZ)
  • Usage Indicator (T=Test, P=Production)
  • Acknowledgment Requested (default: 0)
  • X12 Version: 005010X279A1 (read-only)
Group 4: Transaction Defaults

Defaults applied when CSV doesn't specify per-row values.

  • Default Service Type Code: 30
  • Default Service Date: today
  • Max Batch Size: 5000

Delivery Strategy

📋
Execute as vertical slices rather than directory completion. Deliver working workflows end-to-end before generalizing.
  1. Slice A first: Canonical template intake, row validation, 270 generation, validation of generated output, file download.
  2. Slice B second: 271 upload, tolerant parsing, eligibility dashboard, Excel export.
  3. Generalize after A+B are green: Broaden optional segment coverage, deepen payer-profile seams, expand SNIP depth.
  4. Hardening last: Packaging, OSS ceremony, stricter quality gates once workflow contract is stable.

Design Philosophy: Zero Cognitive Load

The primary user is a non-technical healthcare worker. Every design decision is measured against: "Does this make the user think less?"

1. One Happy Path

User configures provider/payer once in Settings, then uploads files and gets results. Sensible defaults for everything.

2. System Tells You What to Do Next

Every screen has exactly one primary action. After upload: "Preview." After preview: "Generate." The UI is a guided corridor.

3. Errors Are Instructions

Never show "SNIP Level 2 Error: E2.05". Show "Row 12: Member ID '1234' is too short — DC Medicaid requires 8 digits."

4. Auto-Detect, Auto-Correct, Auto-Fill

Auto-detect file type, auto-correct dates, auto-uppercase names, auto-pad member IDs, auto-fill defaults, auto-split at 5000.

5. Progressive Disclosure

Summary first, details on demand. Status badges (Active/Inactive/Error); click to expand. Raw X12 never shown unless requested.

6. Drag-and-Drop Is the Interface

Home page IS a drop zone. Drop any file and the system routes: .xlsx → Generate, .x12+ST01=270 → Validate, .x12+ST01=271 → Parse.
