1. Executive Summary

1.1 What is being built

Three production-grade EDI transaction sets will be added to the existing x12-edi-tools Python library and the FastAPI workbench.

TransactionPurposeDirectionVersion
837IInstitutional claim submission (home-health, hospital, SNF)Outbound (provider → payer)005010X223A2
837PProfessional claim submission (clinician-billed services)Outbound005010X222A1
835Electronic Remittance Advice (payment + adjudication detail)Inbound (payer → provider)005010X221A1

Each is implemented end-to-end: typed Pydantic v2 models → parser → encoder → validator (SNIP 1-5) → payer profile overrides → convenience builder/reader → FastAPI routes. Work is structured so multiple AI agents can operate on non-overlapping files in parallel.

1.2 Why it matters

The library today is eligibility-only (270/271). For a home-health RCM operator, eligibility is a gate, not a revenue event. Claim submission (837I/P) and remittance posting (835) are the backbone of AR, denial management, and cash flow — without them, the product sits adjacent to revenue rather than driving it.

  • 837I — blocking transaction for home-health revenue. Every admission, recertification, and final claim flows through it.
  • 835 — how claims turn into cash and how denials are operationally visible. Posting automation requires this.
  • 837P — unlocks companion lines (physician oversight, therapy evaluations) billed alongside 837I episodes.

1.3 Business value

  • Revenue unlock — first-party submission pipeline without a commercial clearinghouse SDK for envelope construction.
  • Denial intelligence — parsing 835 CAS groups + CARC/RARC codes locally lets us build denial dashboards and payer scorecards without leaking PHI.
  • Posting throughput — structured 835 ingestion compresses an accountant-hour job to an API call.
  • Compliance surface — all SNIP 1-5 validation runs pre-flight, reducing payer rejections and TA1/999 rework.
  • Extensibility — the same scaffolding accommodates 999, 277CA, 276/277, and 278 without re-plumbing envelope, validator, or payer-profile systems.

1.4 Technical scope

In scope:

  • Pydantic v2 domain models for Claim, ClaimLine, Provider, Patient, Payer, Remittance, Adjustment, Acknowledgement, SubmissionBatch.
  • New X12 segments: CLM, CLP, CAS, SVC, HI, SV1, SV2, SV3, HCP, AMT, PWK, CRC, CR1–CR3, CN1, MIA, MOA, DMG/PAT, SBR, OI, MEA, LX, K3, NTE, PLB, BPR, QTY.
  • New loops per implementation guide (2000A/B/C, 2010AA–BB, 2300, 2310A–F, 2320, 2330A–G, 2400, 2410, 2420A–I, 2430, 2440 for 837; 1000A/B, 2000, 2100, 2110 for 835).
  • Transaction classes Transaction837I, Transaction837P, Transaction835 following the 270/271 pattern.
  • Convenience functions build_837i, build_837p, read_835.
  • Parser dispatcher recognizing ST01 ∈ {837, 835} and routing to new loop builders.
  • Encoder extension re-using encode_isa / encode_segment primitives.
  • SNIP 1-5 validator rules per transaction plus payer-profile hook points.
  • DC Medicaid companion-guide override pack plus documented pattern for additional payers.
  • FastAPI routers under /api/v1/claims/* and /api/v1/remittance/*.
  • OpenAPI spec + Swagger UI under docs/api/.
  • Logical ERD generated via eralchemy.
  • Hypothesis property-based tests for roundtrip, parser fuzzing, and control-number invariants.

Out of scope (explicitly):

  • Persistent storage / database adapters — library stays stateless.
  • SFTP / AS2 / trading-partner connectivity — caller's responsibility.
  • Real patient or payer-specific real-world test data — synthetic only per CLAUDE.md.
  • 999, 277CA, 276/277, 278 implementations — scaffolded for, not implemented.
  • UI work in apps/web/ — tracked separately.
  • Background job queues or async claim pipelines.

1.7 Author's opinionated disagreements with the brief

Stated up front so reviewers can overrule before work begins.
  1. "Claim generation" and "claim export / submission preparation" collapse into one endpoint — and 837I and 837P share that endpoint. The library is stateless; generation is the preparation. Separate generate/export endpoints would imply server-side retention. Separate 837I and 837P routes are ceremonial — the request body already carries claim_type as a discriminator and validation dispatches on it internally. Final shape: POST /claims/generate, accepting a discriminated union of InstitutionalClaim and ProfessionalClaim, returning the X12 payload plus a typed envelope manifest. Mixed batches emit separate GS/GE functional groups so each ST uses the correct implementation reference (005010X223A2 vs 005010X222A1). Cost accepted: lose free per-type rate limits / metrics — handled via {txn} metric labels and profile-scoped limits instead.
  2. Reconciliation / posting belongs in the caller's domain. The library projects an 835 into matched/unmatched buckets relative to a caller-supplied list of prior SubmissionBatch control numbers. It does not post or persist.
  3. ERDs for a stateless library are logical, not physical. We use the .er DSL with eralchemy — documentation, not a schema-migration substrate.
  4. Payer-specific logic never lives in transaction builders. Everything payer-specific flows through payers/<payer>/. Inlining payer logic re-creates spaghetti by Q4.

Full tree in the Scaffolding section. Highlights:

  • domain/ — DB-free business objects.
  • models/segments/ — ~35 new segment classes, one file each.
  • models/loops/claims/ and models/loops/remittance/ — loop containers per transaction family.

2.4 Shared infrastructure

  • Envelope construction stays DRY. builders/_common.py extracts ISA/GS/GE/IEA scaffolding. build_270 is refactored in Phase 3 to call it (byte-equal regression test enforced).
  • Parser dispatch is a pure function. parser/transaction_dispatch.py exposes dispatch(st01, implementation_reference) → TransactionLoopBuilder. Unknown ST01 falls back to GenericSegment.
  • Validator layering stays at three tiers: SNIP 1 structural, SNIP 2-3 syntax + code-set, SNIP 4-5 situational + payer.
  • Logging and correlation IDs unchanged. Every new code path uses _logging.build_log_extra with documented non-PHI fields only.

2.5 Payer override strategy

Extend the PayerProfile Protocol with four methods:

class PayerProfile(Protocol):
    name: str

    # Existing
    def validate(self, interchange: Interchange) -> Sequence[ValidationError]: ...
    def get_defaults(self) -> dict[str, object]: ...

    # NEW
    def get_claim_defaults(self, transaction: str) -> dict[str, object]: ...
    def get_remit_overrides(self) -> dict[str, object]: ...
    def validate_claim(self, transaction: Transaction837I | Transaction837P) -> Sequence[ValidationError]: ...
    def validate_remittance(self, transaction: Transaction835) -> Sequence[ValidationError]: ...

Override precedence (highest wins): caller-supplied overrides → payers/<payer>/profile_<txn>.py → payers/base.py defaults → library SNIP defaults.

2.6 Validation strategy

Every transaction is validated at three checkpoints:

  1. Pre-build (domain) — Pydantic validators reject malformed business objects before X12 emission.
  2. Post-build, pre-encode (SNIP 1-3) — validate(interchange, levels=[1,2,3]) against the typed Interchange.
  3. Post-encode or on-parse (SNIP 4-5) — Full validator sweep including payer deltas.

2.7 Data flow

Outbound (claim submission)

POST /api/v1/claims/generate   { claims: [ { claim_type: "INSTITUTIONAL" | "PROFESSIONAL", ... } ] }
  ▼ GenerateClaimRequest (schemas/claims.py; discriminated union on claim_type)
  ▼ services/claims.py::generate_claims_response()
  ▼ builders/claim_837.build_837(claims, config, profile)
       — partitions by claim_type, dispatches to build_837i / build_837p
       — mixed batches emit separate GS/GE groups (one per implementation reference)
  ▼ validator.validate(interchange, levels=[1..5], profile=...)
  ▼ encoder.encode(interchange, config=config, correlation_id=...)
  ▼ wraps x12 + ControlNumbers + ArchiveEntry[]
  ▼ GenerateClaimResponse → client

Inbound (remittance parsing)

POST /api/v1/remittance/835/parse (multipart or inline)
  ▼ services/remittance.py
  ▼ parser.parse(raw, strict=False, on_error="collect")
  ▼ readers/remittance_835.read_835(parse_result)
  ▼ services wraps projection + parse_errors + summary
  ▼ RemittanceParseResponse → client

2.8 Fit alongside existing 270/271 support

Purely additive. No existing public API changes. build_270 / read_271 / parse / encode / validate signatures are preserved. New public functions join __init__ in Phase 7.

3.3 Naming conventions

  • Segments: models/segments/<lowercase_segment_id>.py; class <UpperCase>Segment; one segment per file.
  • Loops: models/loops/<family>/loop_<numeric_id>[_<scope>].py; class Loop<NumericId>_<TxnSuffix>.
  • Transactions: transaction_<txn_lower>.py; class Transaction<TxnUpper>.
  • Builders: builders/claim_<txn_lower>.py; function build_<txn_lower>.
  • Readers: readers/<scope>_<txn>.py; function read_<txn>.
  • Domain objects: Singular, business-language. Claim, not Claim837I. The wire format is implementation detail.
  • Validator rules: error codes X12-<TXN>-<NNN>.
  • Payer overrides: payers/<slug>/profile_<txn>.py, constants_<family>.py.
  • Fixtures: tests/fixtures/<domain>/<txn>/synthetic_<scenario>.x12.

3.4 Code organization strategy

  • One concept per file. Don't bundle SV1 and SV2.
  • No cross-file enum leakage. Shared enums move to common/enums.py only when ≥ 2 segments consume them.
  • No inline loop definitions in transaction files. Transaction modules stay under ~80 lines.
  • No helper dumping grounds. _common.py is for envelope construction only.
  • Strict typing throughout. mypy --strict enforced in CI.

  • Every payer is a sibling under payers/, owned by one agent.

  • Constants files hold payer-scoped code sets.

3.8 How scaffolding prevents agent collision

  1. One file per agent per task. Phase 3 assigns claim_837i.py to Agent D and claim_837p.py to Agent E; they do not touch each other's files.
  2. __init__.py public-surface edits only in Phases 0 and 7.
  3. Segment and loop files are write-once. Missing segments route through Phase 2 patches, not Phase 3 reach-ins.
  4. Per-agent worktrees. git worktree add ../X12-837I-worktree dev.
  5. Fixture isolation. Phase 8 agents create fixtures under their transaction subfolder only.
  6. No cross-layer reach-through. Routers import services; services import library public surface; routers never import x12_edi_tools.models.segments.clm. Lint-enforced.
  7. Deterministic __all__ ordering. Alphabetical. Mechanical.

Appendix C — Glossary

  • Interchange — ISA/IEA-wrapped outer envelope containing one or more functional groups.
  • Functional group — GS/GE-wrapped group of same-ST01 transactions.
  • Transaction — ST/SE-wrapped payload (e.g., one 837I).
  • Loop — Ordered segment group within a transaction (e.g., Loop 2300 Claim Information).
  • SNIP — Strategic National Implementation Process five-level validation taxonomy.
  • TOB — Type of Bill; four-digit institutional billing code.
  • POS — Place of Service; two-digit professional service code.
  • CARC — Claim Adjustment Reason Code (CAS02, 835).
  • RARC — Remittance Advice Remark Code (LQ02, 835).
  • PLB — Provider Level Adjustment (835).
  • PSC — Payer Specific Companion guide.
